# CRM 入门：本系统做了什么、指令怎么串、AI 和记忆在哪

> 写给 CRM 新手：先懂业务，再懂指令，最后懂技术分层。  
> 演示口播可配合 [DEMO.md](./DEMO.md)。

---

## 1. CRM 是什么（30 秒）

**CRM（Customer Relationship Management，客户关系管理）** 帮企业管三件事：

1. **从哪来客户**（渠道、活动、线索）
2. **怎么跟进成交**（客户、商机、跟进记录）
3. **效果怎么样**（来源分析、漏斗、面板）

本系统把上面这些事做成：**在飞书里发指令 → 数据写入数据库 → 随时查分析**。  
不是「聊完就忘」的 ChatGPT，而是**有记忆的业务系统**。

---

## 2. 本系统实现了哪些 CRM 功能

按业务模块划分（不是按技术模块）：

| CRM 模块 | 本系统做了什么 | 典型使用者 |
|----------|----------------|------------|
| **获客 / 渠道** | 记录渠道名称、类型、成本；看各渠道线索量与质量 | 市场、运营 |
| **营销活动** | 创建活动、绑定渠道、设预算与目标 | 市场 |
| **线索（Lead）** | 录入潜在客户；**自动评分**；列表按分排序 | 销售、市场 |
| **客户（Customer）** | 从线索转化或直接创建；归属负责人 | 销售 |
| **商机（Opportunity）** | 线索转客户时自动建初始商机（阶段、金额） | 销售 |
| **跟进（Follow-up）** | 记录沟通内容；生成简短摘要与「下一步」建议日期 | 销售 |
| **分析 / 看板** | 获客分析、来源分析、漏斗、个人面板 | 销售、管理者 |

**当前没做主线：** 复杂工单中心、合同审批、权限审计、独立 Web 后台。

## 3. 指令 ↔ 功能对照表

飞书里发给 CRM 机器人的内容，**推荐用自然语言**；工作流内 LLM 会先解析成 `/指令` 再执行业务。仍兼容直接发斜杠指令。

### 3.0 自然语言示例（推荐演示用）

| 你说 | 系统解析为 | 功能 |
|------|------------|------|
| 帮我看看各渠道获客效果 | `/获客分析` | 渠道分析 |
| 列出当前线索 | `/线索列表` | 线索列表 |
| 录入线索：周总，云启科技，官网，高意向，20万，跟进不及时 | `/线索创建 …` | 新建线索+评分 |
| 把线索 3 转成客户 | `/线索转客户 3` | 转客户+商机 |
| 跟进一下周总，他下周想看演示 | `/跟进 客户:周总 内容:…` | 记跟进 |

### 3.1 获客营销（斜杠指令，可选）

| 指令 | 做什么 | 数据变化 |
|------|--------|----------|
| `/渠道创建 名称:… 类型:… 成本:…` | 新增一条获客渠道 | `channels` 表 |
| `/活动创建 名称:… 渠道:… 预算:… 目标:…` | 新增营销活动，可绑定渠道 | `campaigns` 表 |
| `/线索创建 名称:… 公司:… 来源:… 意向:… 预算:… 痛点:…` | 录入线索并**自动评分** | `leads` 表 |
| `/线索列表` | 按评分看线索 | 只读 |
| `/线索评分 {id}` | 对某条线索重新算分 | 更新 `leads.score` |
| `/线索转客户 {id}` | 线索变客户，并建一条商机 | `customers` + `opportunities`，线索状态变 converted |
| `/来源分析` | 各**来源**线索数、转化、均分、潜在金额 | 只读统计 |
| `/获客分析` | 各**渠道**线索数、转化、成本、均分 + 一句建议 | 只读统计 |

### 3.2 客户与销售

| 指令 | 做什么 | 数据变化 |
|------|--------|----------|
| `/客户创建 名称:… 电话:… 来源:…` | 不经过线索，直接建客户 | `customers` 表 |
| `/我的客户` | 看当前负责人名下的客户 | 只读 |
| `/跟进 客户:… 内容:…` | 记一条跟进 | `followups` 表 |
| `/漏斗` | 线索状态 + 商机阶段分布与解读 | 只读 |

### 3.3 辅助

| 指令 | 做什么 |
|------|--------|
| `/help` | 指令菜单 |
| `/面板` | 待跟进线索数、高分线索、待办、获客建议 |

参数写法：`名称:值` 用空格隔开多个参数，例如：

```text
/线索创建 名称:周总 公司:云启科技 来源:官网表单 意向:高 预算:20万 痛点:销售跟进不及时
```

## 4. 指令怎么「接龙」——推荐业务链路

把 CRM 想成一条流水线，**上一步的输出是下一步的输入**。

### 4.1 主线：从获客到成交（演示最好用这条）

```text
/获客分析
    ↓ 了解哪条渠道值得投
/线索列表
    ↓ 看现有高分线索（如「张明」）
/线索创建 …
    ↓ 新线索进库 + 自动评分
/线索列表
    ↓ 确认新线索 ID
/线索转客户 {id}
    ↓ 产生「客户」+「商机」
/跟进 客户:周总 内容:…
    ↓ 销售过程留痕
/漏斗  或  /面板
    ↓ 管理者看阶段与待办
```

**讲解要点：** 这不是 7 个无关命令，而是 **市场获客 → 销售跟进 → 管理看数** 的闭环。

### 4.2 支线 A：只做市场录入（不立刻转客户）

```text
/渠道创建 …  →  /活动创建 …  →  /线索创建 …  →  /来源分析
```

适合讲：「活动结束后再统一看来源效果」。

### 4.3 支线 B：已有客户，直接跟进

```text
/客户创建 …  →  /我的客户  →  /跟进 …
```

适合讲：「转介绍客户不经过线索表也可以进系统」。

### 4.4 随时可用的「读操作」

这些**不改变数据**，可在任意时刻插入：

```text
/help  /获客分析  /来源分析  /线索列表  /漏斗  /面板
```

## 5. 从系统设计讲：CRM 里 AI 能做什么

分三层说，听众容易懂：

### 5.1 传统 CRM（无 AI）

- 表单录入、列表查询、固定报表  
- **本系统已具备**：指令录入、MySQL 持久化、获客/漏斗统计  

### 5.2 本系统**已经有的「轻量智能」**（不是大模型，是规则引擎）

| 能力 | 实现方式 | 对应指令/现象 |
|------|----------|----------------|
| **线索自动评分** | 规则：来源、意向、预算、痛点关键词 | `/线索创建`、`/线索评分` |
| **评分原因说明** | 规则拼接成可读句子 | 创建线索后的「原因：…」 |
| **跟进摘要** | 截取/整理原文 | `/跟进` 返回的「摘要」 |
| **下一步建议** | 关键词（报价/演示/预算）→ 固定话术 | `/跟进` 的「下一步」 |
| **渠道/漏斗解读** | 简单规则（如「某渠道线索最多」） | `/获客分析`、`/漏斗` |

**口播：** 「AI 不一定是 ChatGPT；**可解释的评分和提醒**也是 CRM 智能化的一部分。」

### 5.3 **已接入的 AI（Dify LLM）**

| 能力 | 实现方式 | 例子 |
|------|----------|------|
| **自然语言 → CRM 指令** | Dify 工作流 `自然语言转指令` 节点（LLM） | 「看看获客效果」→ `/获客分析` |

### 5.4 **规划中的 AI**（RAG / 更深推理）

| 能力 | 适合用什么 | 例子 |
|------|------------|------|
| 自然语言代替 `/指令` | Dify LLM 意图识别 | **已接入** `crm_main` 工作流 |
| 跟进话术、竞品应对 | **RAG 知识库** | 「这类客户怎么报价？」 |
| 标签/阶段建议 | LLM + **MySQL 查现状** | 查线索后再生成「建议推进到方案报价」 |
| 业绩解读 | LLM + 统计 API | 「本月漏斗卡在哪？」 |

**口播：** 「结构化事实查 MySQL，非结构化经验查知识库，大模型负责**组织成建议**——这是 CRM + AI 的常见分工。」

## 6. 记忆：MySQL 和 RAG（Embedding）分别管什么

### 6.1 结论先说

| 类型 | 技术 | 本系统**当前是否在用** | 存什么 |
|------|------|------------------------|--------|
| **结构化记忆** | **MySQL** `dify_crm` | **是，主记忆** | 线索、客户、商机、金额、状态、跟进时间 |
| **向量记忆（RAG）** | Dify 知识库 + **Embedding** + **Weaviate** | **否，尚未接入 crm_main 工作流** | 话术、FAQ、案例、政策（规划） |

对话本身**不会**自动变成长期记忆；只有走 CRM 指令/API 写入 MySQL 的才会留下。

### 6.2 MySQL ——「精确记忆」

- 库名：`dify_crm`  
- 核心表：`channels` → `campaigns` → `leads` → `customers` → `opportunities` → `followups`  
- 特点：可 COUNT、可 JOIN、可排序（如按评分）  
- **适合：** 「线索 #3 评分多少？」「官网表单来了几条线索？」  

演示时可说：**「这是 CRM 的账本，AI 不能瞎编，要以这里为准。」**

### 6.3 RAG / Embedding ——「经验记忆」（规划层）

- **Embedding**：把文档切成片段，变成向量，语义相似检索  
- **Weaviate**：Dify 自托管部署里已有的向量库（在 `DifyDeploy` Docker 中）  
- **适合进 RAG 的：** 产品手册、销售话术、FAQ、成功案例（见 PRD）  
- **不适合进 RAG 的：** 客户电话、商机金额、线索状态（应查 MySQL）  

**当前 `crm_main` 工作流：** 只有「收消息 → 调 CRM API → 回 summary」，**没有**知识库检索节点，也**没有**调用 Embedding。  
表 `knowledge_assets` 里预置了几条「该进知识库」的文本说明，供以后导入 Dify 知识库用。

### 6.4 混合场景（将来演示时可讲）

用户问：「周总这条线索，下一步怎么跟进？」

1. **MySQL**：查周总来源、评分、预算、最近跟进  
2. **RAG**：查「高意向 B2B 跟进话术」「竞品对比」  
3. **LLM**：合并成一段建议

## 7. 技术栈（介绍末尾 1 分钟可选）

```text
飞书 → LangBot（长连接）→ Dify 工作流 crm_main → DifyCRM API → MySQL dify_crm
```

| 组件 | 作用 |
|------|------|
| **飞书** | 用户入口、协作 |
| **LangBot** | 接飞书消息，调 Dify |
| **Dify** | 工作流编排（后续 LLM/RAG） |
| **DifyCRM API** | 业务规则、指令解析、统计 |
| **MySQL** | 业务数据真源 |
| **Weaviate** | Dify 知识库向量存储（RAG 用时才参与 CRM 链路） |

与旧 **表格分析机器人** 独立：不同 LangBot 流水线、不同 Dify 应用，互不影响。

## 8. 10 分钟介绍时可用的「三层叙事」

1. **业务层（3 分钟）**  
   CRM 管获客、跟进、分析；本系统用飞书指令完成。  
   走一遍：获客分析 → 线索创建 → 转客户 → 跟进 → 漏斗。

2. **AI 层（3 分钟）**  
   已有：自动评分、跟进建议、分析解读（规则型）。  
   规划：LLM 理解自然语言 + RAG 话术库；结构化数据仍查 MySQL。

3. **技术/记忆层（2 分钟）**  
   MySQL = 账本；RAG/Embedding = 经验库（下一步接）。  
   一体化架构，不是多个系统拼凑。

4. **Q&A（2 分钟）**
